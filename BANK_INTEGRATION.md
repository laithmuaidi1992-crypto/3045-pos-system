# دليل دمج جهاز POS البنك العربي · 3045 سوبر ماركت

> طبقة قابلة للتبديل لربط نظام نقاط البيع مع جهاز **WizarPOS Q2** التابع للبنك العربي الأردني.

## نظرة عامة على البنية

```
┌─────────────────────────┐         ┌────────────────────────┐         ┌──────────────────┐
│  متصفح الكاشير          │         │  خدمة الجسر المحلية     │         │  جهاز WizarPOS Q2 │
│  (3045 POS)             │ ──HTTP──→│  (Node.js · :8765)     │ ──TCP──→│  192.168.1.X      │
│  React + Vite           │  JSON    │  bridge/server.js      │ JSON-LN │  Arab Bank app    │
└─────────────────────────┘         └────────────────────────┘         └──────────────────┘
```

النظام يعمل بثلاث أوضاع قابلة للتبديل من تبويب **الإعدادات → دمج جهاز الدفع البنكي**:

| الوضع | كيف يشتغل | متى نستخدمه |
|------|---------|------------|
| **بدون دمج (off)** | السلوك القديم — مودال "بانتظار فيزا" + زر تأكيد فقط | حالة افتراضية أو عند تعطيل الدمج |
| **يدوي (manual)** | بعد الدفع على الجهاز، الكاشير يدخل RRN + رمز الموافقة في النظام | يشتغل **اليوم** بدون أي دعم تقني من البنك |
| **جسر مباشر (bridge)** | النظام يرسل المبلغ تلقائياً للجهاز، يعرض الحالة الحية، يستلم النتيجة | يحتاج وثائق ECR من البنك + تشغيل خدمة الجسر |

## التثبيت — خطوة بخطوة

### 1. ترحيل قاعدة البيانات

افتح Supabase SQL Editor ونفّذ الملف:

```
migrations/2026_04_25_bank_terminal_fields.sql
```

هذا يضيف الأعمدة التالية لجدول `transactions`:
- `bank_rrn` — رقم العملية المرجعي
- `bank_approval` — رمز الموافقة
- `bank_last4` — آخر 4 أرقام من البطاقة
- `bank_brand` — نوع البطاقة (Visa/MC/JoMoPay)
- `bank_terminal_id` — معرّف الجهاز
- `bank_response` — استجابة الجهاز كاملة (JSON)

ويُنشئ جدول جديد `bank_settlement` لاستيراد تقرير التسوية اليومي من البنك ومطابقته مع المبيعات.

### 2. تفعيل الوضع اليدوي (يشتغل فوراً)

1. ادخل النظام كمدير → **الإعدادات**
2. ابحث عن قسم **💳 دمج جهاز الدفع البنكي**
3. اختر **يدوي**
4. أدخل **معرّف الجهاز** (موجود على ظهر جهاز Q2 — رقم تسلسلي)
5. اضغط **حفظ**

من الآن: عند كل عملية دفع بفيزا، النظام سيطلب من الكاشير:
- رقم العملية (RRN) — مكتوب على إيصال الجهاز
- رمز الموافقة (Approval Code) — مكتوب على إيصال الجهاز
- آخر 4 أرقام من البطاقة (اختياري)
- نوع البطاقة (اختياري)

ويحفظها مع الفاتورة. هذه البيانات تظهر في **سجل المبيعات** ويمكن استخدامها لاحقاً لمطابقة تقرير البنك اليومي.

### 3. تفعيل الجسر المباشر (يحتاج وثائق البنك)

#### أ. تشغيل خدمة الجسر على جهاز الكاشير

افتح موجه الأوامر (Terminal/CMD) في مجلد المشروع:

```bash
cd bridge
npm install     # المرة الأولى فقط (لا توجد حالياً تبعيات خارجية)
node server.js
```

سترى:
```
─────────────────────────────────────────────────────────────
  3045 POS · Bank Terminal Bridge
  Listening on http://127.0.0.1:8765
  Driver: wizarpos-q2-simulated
─────────────────────────────────────────────────────────────
```

> **ملاحظة:** الخدمة تعمل افتراضياً في **وضع المحاكاة** — ترجع استجابات وهمية مقبولة لاختبار الواجهة. لاستخدامها مع جهاز حقيقي، انظر القسم التالي.

#### ب. تفعيل الوضع في النظام

1. الإعدادات → دمج جهاز الدفع البنكي
2. اختر **جسر مباشر**
3. عنوان خدمة الجسر: `http://localhost:8765` (افتراضي)
4. اضغط **🩺 اختبار الاتصال** — يجب أن يظهر "متصل"
5. اضغط **🧪 عملية تجريبية** — يُرجع RRN ورمز موافقة وهميَّين للتأكد من سريان الإشارة عبر الواجهة
6. **حفظ**

## ربط جهاز WizarPOS Q2 الفعلي

الـ Q2 جهاز ذكي يعمل بنظام Android. وضع المحاكاة الحالي في `bridge/wizarpos-driver.js` يحتاج استبداله بكود يتحدث مع الجهاز الفعلي. الخيارات الواقعية:

### الخيار الأول: TCP عبر WiFi (الأكثر شيوعاً)

معظم نشرات Q2 من البنوك في الأردن تشغّل على الجهاز تطبيق "ECR Listener" على منفذ TCP محدد (غالباً 8888 أو 1234 أو 6000). الكود الحالي في `wizarpos-driver.js` يحتوي على قالب جاهز لهذا الوضع — ما تحتاجه:

1. **عنوان IP الجهاز** — اطلب من البنك حجزه ثابتاً في راوتر المتجر
2. **رقم المنفذ** — يحدده البنك حسب نشرته
3. **بروتوكول الرسائل** — هل JSON-line أم XML أم ISO8583-lite؟ ما هي أسماء الحقول؟

عدّل المتغيرات في أعلى `wizarpos-driver.js`:
```javascript
const TERMINAL_IP = "192.168.1.100";   // ← IP الجهاز
const TERMINAL_PORT = 8888;             // ← المنفذ المعتمد من البنك
const SIMULATE = false;                 // ← أوقف المحاكاة
```

ثم أعدِّل أسماء الحقول في `tcpCharge()` لتطابق وثائق البنك (مثلاً `responseCode` قد تكون `result_code`، و `rrn` قد تكون `referenceNumber`).

### الخيار الثاني: Bluetooth أو USB

إذا كانت نشرة البنك تتطلب اتصالاً بكابل أو بلوتوث، نحتاج لإضافة:
```bash
npm install serialport
```
ثم إضافة Driver جديد على غرار `tcpCharge` يستخدم منفذ COM.

### الخيار الثالث: Cloud API (نادر في الأردن)

بعض البنوك تستخدم نموذجاً يدفع فيه النظام طلب الدفع لـ API سحابي يُحوّله لجهاز Q2 عبر FCM. يلزم API key + endpoint من البنك.

2. **البروتوكول:** TCP/Serial/Bluetooth/Cloud؟
3. **منفذ التواصل:** أي بورت TCP؟
4. **تنسيق الرسائل:** JSON أم XML أم ISO8583؟ مع وثيقة تحدد الحقول
5. **ملف اختبار:** Merchant ID + Terminal ID للاختبار بدون رسوم
6. **الإلغاء (Void/Refund):** هل مدعوم عبر ECR؟ ما هي الرسائل؟
7. **التسوية اليومية:** كيف تُسحب تقارير التسوية؟ ملف CSV/Excel/API؟

## مطابقة التسوية اليومية (Reconciliation)

بعد كل يوم عمل، البنك يُصدر تقرير تسوية يحتوي على كل العمليات الناجحة. نظامك الآن يحفظ نفس الحقول (RRN, Approval Code, Last4, Brand) — مما يُمكّن مطابقة دقيقة.

ميزة المطابقة الكاملة (استيراد ملف CSV من البنك ومقارنته بمبيعات النظام تلقائياً) يمكن إضافتها كخطوة ثانية بعد ما الدمج الأساسي يستقر. الـ schema جاهز (جدول `bank_settlement` موجود).

## استكشاف الأخطاء

### "خدمة الجسر غير متاحة" في الإعدادات
- تأكد أن `node server.js` ما زال يعمل في موجه أوامر مفتوح
- تأكد أن المنفذ 8765 ليس محجوزاً من برنامج آخر
- في Windows: قد يحتاج جدار الحماية للسماح للمنفذ — `netsh advfirewall firewall add rule name="3045 Bridge" dir=in action=allow protocol=TCP localport=8765`

### "Cannot reach terminal at 192.168.1.X:8888"
- تأكد جهاز Q2 على نفس شبكة WiFi
- تأكد أن IP الجهاز ثابت (DHCP Reservation في الراوتر)
- اتصل بالبنك للتأكد من تفعيل ECR Listener على الجهاز

### "Invalid response from terminal"
- بروتوكول البنك مختلف عمّا في القالب — راجع وثائق ECR وعدّل `tcpCharge()` في `wizarpos-driver.js`

### الكاشير ينقر "تأكيد" قبل تأكيد الجهاز (في وضع الجسر)
- لا يحدث — النظام يقفل زر التأكيد تلقائياً حتى يستلم استجابة موافقة من الجسر

## ملخص الحقول المضافة لقاعدة البيانات

| الحقل | النوع | الوصف |
|------|------|------|
| `bank_rrn` | text | Retrieval Reference Number — معرّف فريد للعملية |
| `bank_approval` | text | رمز الموافقة من البنك المُصدر |
| `bank_last4` | text | آخر 4 أرقام من البطاقة (آمن PCI) |
| `bank_brand` | text | Visa / MasterCard / JoMoPay / Mada / AmEx |
| `bank_terminal_id` | text | معرّف الجهاز الفعلي |
| `bank_response` | jsonb | استجابة الجهاز كاملة (للتدقيق) |

## خطوات تالية مقترحة

1. **اليوم:** فعّل الوضع اليدوي — يبدأ التقاط RRN + Approval فوراً
2. **أسبوع 1:** اتصل بالبنك العربي واحصل على وثائق ECR + ملف تجريبي
3. **أسبوع 2:** عدّل `wizarpos-driver.js` حسب الوثائق وفعّل وضع الجسر بالاختبار
4. **أسبوع 3:** انقل الإنتاج إلى وضع الجسر، احتفظ بالوضع اليدوي كـ fallback
5. **شهر 2:** أضف ميزة استيراد تقرير التسوية اليومي ومطابقته أوتوماتيكياً
